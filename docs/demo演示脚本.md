# AI 思政教育视频工作流平台 Demo 演示脚本

## 1. 演示目标

本次 demo 的目标不是展示“单点 AI 能力”，而是展示一套适合学校落地的 `AI 思政视频生产平台`：

- 教师可以快速发起并审核一支课堂视频
- 学生可以参与创作，但不拥有审稿权和终审权
- 学校管理员可以维护本校素材、品牌和标签
- 平台运营可以跨校做安全巡检、规则调权和能力治理
- 整个平台具备 `前置审核 + 后置审核 + 审计留痕 + 多学校隔离 + 工作流编排`

当前 demo 全部基于本地 mock 数据，不依赖真实后端。

## 3. 推荐演示顺序

建议总时长控制在 `12 ~ 18 分钟`。

### 第一段：开场总览，2 分钟

登录教师账号后，先进入 `MVP 总览` 页面。

这一段重点讲 4 件事：

1. 这不是单个生成工具，而是一套教学场景工作流平台。
2. 主流程是：`发起 -> 脚本生成 -> 教师强制审稿 -> 批量生成 -> 微调 -> 输出复审 -> 视频入库`。
3. 平台把 `角色权限、学校隔离、安全审核、工作流编排` 放在同一套系统里。
4. 首页已经把 `产品价值、非功能约束、技术架构、实施路径` 汇总展示出来。

推荐话术：

> 我们不是只演示“AI 能生成视频”，而是演示“学校敢不敢用、老师能不能管、平台能不能长期运营”。

### 第二段：教师主流程，4 到 5 分钟

从教师视角依次演示：

1. `创作发起台`
2. `脚本审核台`
3. `生成队列`
4. `安全复审`
5. `视频库审计`

#### 2.1 创作发起台

重点看：

- 可以输入自然语言指令
- 可以选择学段、教材版本、模板、学校策略、知识点标签
- 页面右侧会实时生成“最终提示词预览”
- 同时能看到输入审核关注点

要强调：

- 年级会自动影响语速、语言层级和默认画风
- 学校素材会自动参与提示词注入
- 学生和教师发起同一个任务，后续审核责任不同

推荐话术：

> 这里体现的是“内容发起标准化”，老师不需要自己拼 prompt，系统会自动注入课程大纲、学校素材和年级参数。

#### 2.2 脚本审核台

重点操作：

- 选中任意镜头
- 修改一段旁白
- 说明“只改旁白只重配音，不重绘画面”
- 再修改一段画面描述
- 说明“改画面就会重绘并重配音”

要强调：

- 教师必须确认脚本后，系统才能进入批量生成
- 这个环节是平台的核心教学安全阀
- 不是黑箱自动出片，而是“教师主导、AI 辅助”

推荐话术：

> 这一页最关键，因为它把老师从“被动看片人”变成“工作流里的决策人”。

#### 2.3 生成队列

重点看：

- 任务状态区分为 `draft / generating / review / approved / blocked`
- 页面能看到排队信息、任务步骤、镜头运行态、任务时间线
- 教师能看到“仅重配音”和“重绘画面并重配音”两个动作

要强调：

- 高峰期不是卡死，而是有排队位次和预计等待时间
- 所有动作都写入时间线，方便审计
- 学生账号进入这里时，只能重提，不能放行

推荐话术：

> 这页主要回答一个现实问题：学校高峰期同时用的时候，系统如何稳住。

#### 2.4 安全复审

重点看：

- 展示“历史人物强制复审”的案例
- 展示“学生输入违规被前置拦截”的案例
- 切换文本、图像、视频、音频四个审校维度
- 演示不同处理动作：`教师通过 / 要求重新生成 / 退回修改`

要强调：

- 输入阶段和输出阶段都有审核
- 不是只有“通过/不通过”两种结果，而是有结构化的处理动作
- 教师端能看到原因，平台运营端能跨校巡检

推荐话术：

> 我们把审核做成“可解释、可追踪、可复核”的过程，而不是一个黑盒分数。

#### 2.5 视频库审计

重点看：

- 教师作品和学生作品的可见性不同
- 视频进入视频库后仍保留完整审计链路
- 有课堂使用方式、留存策略、限制条件

要强调：

- 学生作品不是生成完就可见，必须经过教师确认
- 平台明确限制“仅限校内教学使用”
- 每个版本的关键动作都能回溯

## 4. 第三段：学生视角，2 分钟

切换到 `student` 账号。

建议演示：

1. 首页卡片会明显变化
2. `创作发起台` 里默认发起角色会变成学生
3. `生成队列` 中可以看到“学生违规指令待重提”案例
4. 说明学生只能修改后重新提交，不能审稿和终审

重点话术：

> 学生被允许参与创作，但平台明确保留教师的审核和放行责任，这样才能适应中小学场景。

## 5. 第四段：学校管理员视角，2 到 3 分钟

切换到 `admin` 账号。

建议演示：

1. `学校素材与标签`
2. `审核规则管理`
3. `角色与权限中心`

### 重点讲什么

- 校徽、校训、校园背景图都按学校隔离
- 切换学校后，主题色、系统名、水印同步变化
- 学校管理员主要维护学校素材和查看学校规则
- 学校管理员不负责跨校治理

推荐话术：

> 这部分体现的是平台的“学校接入能力”，不同学校接进来后，不需要重做一套系统。

## 6. 第五段：平台运营视角，2 到 3 分钟

切换到 `operator` 账号。

建议演示：

1. `安全复审`
2. `审核规则管理`
3. `配额、队列、成本与技术路线`

### 重点讲什么

- 平台运营可以跨校查看复审案例
- 可以调节规则权重并用模拟器看结果变化
- 可以从运营页讲清楚后端技术路线、成本结构和扩展计划

推荐话术：

> 如果说教师页面讲的是“能不能用”，平台运营页面讲的就是“能不能规模化管理”。

## 10. 常见问答建议

### Q1：这套系统现在是不是已经接真实模型了？

建议回答：

> 当前 demo 全部使用本地 mock 数据，目的主要是完整展示产品流程、权限、安全和运营能力。真实模型与后端编排能力已经在技术方案里规划清楚，后续可以按阶段接入。

### Q2：为什么一定要有教师强制审稿？

建议回答：

> 因为中小学场景里，老师必须保留内容确认责任。平台的设计重点不是“完全自动”，而是“在保证教学安全前提下，把老师的制作效率拉起来”。

### Q3：多学校接入时，怎么保证不会串素材？

建议回答：

> 我们把学校素材、品牌主题、学校规则都按学校 ID 做了隔离。现场切换学校时，系统名称、颜色、素材都会同步变化，这就是为了说明平台具备多学校运营能力。

### Q4：高峰期并发怎么办？

建议回答：

> 平台从 MVP 阶段就把队列和状态机放进设计里，不是请求卡死，而是进入可视化排队。用户能看到排队位置和预计等待时间，平台能稳定处理高峰请求。

## 11. 结尾建议

最后建议用一句话收尾：

> 这个项目的核心价值，不只是“AI 生成了一个视频”，而是把学校真正需要的“创作、审核、发布、治理、运营”做成了一套能落地的平台。
