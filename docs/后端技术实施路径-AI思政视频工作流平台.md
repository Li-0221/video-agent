# AI+思政教育视频工作流平台后端技术实施方案

## 1. 方案摘要

本项目后端采用 `Python + FastAPI` 作为统一技术底座，在 MVP 阶段以 `Docker` 方式完成部署，优先保障快速上线、稳定交付和较低运维门槛；在业务规模扩大后，再逐步演进为 `统一接入网关 + 分布式服务 + K8s` 架构，以支撑更高并发、更多学校接入和更复杂的工作流编排需求。

平台后端的建设重点，不是单纯接入某一个视频生成模型，而是形成一套 `可审核、可追踪、可扩展、可长期运营` 的教育视频生产工作流。面向教师、学生、学校管理员和平台运营四类角色，后端需要统一承载内容生成、审核中台、学校资产管理、权限控制、任务排队和审计留痕等能力，最终形成适合教育场景落地的平台型产品，而不是单点工具。

在生成侧，以 `Seedance 2.0` 作为视频生成主能力；在安全侧，以 `腾讯云文本内容安全 TMS + 腾讯云数据万象 CI` 作为审核主通道，并在平台内部建设一层“审核规则中台”，实现平台级与学校级规则配置、风险分级、人工复审流转和审核明细输出。整体实施路线重点突出三点：

1. 初期以单体增强型后端完成 MVP 闭环，优先保证交付速度和可控性。
2. 审核同时前置到输入阶段、后置到成片阶段，形成双重安全保障。
3. 所有生成与审核统一纳入任务编排体系，确保高峰期系统可控、可排队、可追踪。

## 2. 总体技术路线

### 2.1 技术选型

- Web API 框架：`FastAPI`
- 数据库：`PostgreSQL`
- 缓存与队列：`Redis`
- 异步任务：`Celery` 或 `ARQ`
- 对象存储：初期可兼容 `腾讯云 COS`，开发环境可用 `MinIO`
- 媒体处理：`FFmpeg`
- ORM 与迁移：`SQLAlchemy + Alembic`
- 鉴权：基于登录令牌、角色权限和数据范围控制
- 可观测性：监控、日志与告警体系

### 2.2 架构设计

结合当前阶段，MVP 建议采用“统一应用服务 + 异步任务处理”的模式：

- `API 服务`：处理登录、任务发起、脚本编辑、审核结果查询、管理后台接口
- `工作流编排服务`：管理视频任务状态机、子任务拆分、失败重试、回调处理
- `生成适配层`：封装 Seedance 2.0、TTS、字幕、转码、封面生成等外部能力
- `审核中台`：统一封装腾讯云 TMS/CI，自定义规则和人工复审流转
- `资产中心`：管理学校素材、模板、知识点标签、输出视频与审计附件

这一方案的优势在于：上线速度快、维护成本相对较低、接口边界清晰，并且后续可以平滑拆分出“生成编排”“审核中台”“权限中心”等独立服务，具备良好的演进空间。

## 3. 核心后端模块拆分

### 3.1 账号、组织与权限中心

后端虽然采用统一系统，但必须支持清晰的角色视图与数据权限边界：

- 学生：只能发起自己的任务、查看自己的作品、查看被退回原因
- 教师：可审核脚本、微调镜头、审批学生作品、查看本班或本校授权范围内违规记录
- 学校管理员：可管理学校素材、标签、默认参数、查看本校全量记录
- 平台运营：可管理全局审核策略、风险阈值、标签库、异常任务与跨校审计

权限模型采用“角色权限 + 数据范围控制”：

- 角色权限控制菜单、接口和操作权限
- `数据范围控制` 控制能看到哪所学校、哪个班级、哪些任务和哪些审核记录

这样既可以实现“同一个后端、不同角色看到不同板块”的产品效果，也可以避免为不同角色重复建设多套系统。

### 3.2 视频工作流中心

视频工作流本质上不是单一任务，而是一组可追踪、可编排的子任务。主任务可拆分为：

- 输入审核任务：对用户提交的主题、补充说明、标签注入后的最终生成指令进行审核，提前拦截违规内容。
- 脚本生成任务：根据教师或学生输入内容，自动生成结构化脚本，包括分镜、旁白、画面描述和预计时长。
- 教师审核确认节点：由教师对脚本进行确认、修改或退回，未确认前不进入后续素材生成阶段。
- 分镜素材生成任务：按照确认后的脚本逐镜生成画面素材，支持文生图或文生视频的生成方式。
- 音频与字幕合成任务：对每个镜头生成配音、字幕和必要的音轨内容，并与分镜素材进行对应。
- 成片合成任务：将各分镜素材、字幕、配音、背景音乐及校徽校训等元素进行统一拼接，生成最终视频文件。
- 输出审核任务：对合成后的字幕、画面、音频和完整视频做最终审核，识别违规风险和疑似风险。
- 人工复审任务：对命中规则或处于疑似风险区间的视频，转入教师或运营侧进行人工确认。
- 发布入库任务：将审核通过的视频写入视频库，供教师播放、下载或在学生端按权限查看。

每个任务都要有明确状态，例如：

- `pending`
- `running`
- `waiting_review`
- `review_rejected`
- `blocked`
- `completed`
- `failed`

通过这种方式，前端才能稳定展示“排队中、生成中、待教师确认、待复审、已通过、已拦截”等关键状态，增强整个流程的可视化与可管理性。

### 3.3 生成适配层

生成能力通过统一适配层进行封装，避免业务流程与单一厂商深度绑定：

- 脚本生成：文本模型
- 分镜图/视频生成：`Seedance 2.0`
- 配音：TTS 服务
- 字幕：ASR/文本切分服务
- 转码与拼接：FFmpeg

Seedance 2.0 在平台中作为主视频生成引擎，后端通过统一能力接口进行接入：

- `generate_storyboard_assets`
- `regenerate_scene_visual`
- `synthesize_voice`
- `compose_video`

这样在后续增加备选模型、调整路由策略或更换供应商时，不需要重写核心业务流程。

### 3.4 审核中台

审核中台是平台中的关键能力模块，作为独立模块建设。它不仅承担第三方审核接口接入，还负责：

- 审核策略路由
- 规则配置与版本管理
- 第三方审核结果标准化
- 风险分级
- 人工复审分发
- 审核明细留存与导出

平台最终返回给前端的结果，不应只是“通过/不通过”，而应是结构化审核结果：

- 命中的审核阶段
- 风险标签
- 置信度/分值
- 命中的字幕片段、关键帧或音频片段
- 触发的自定义规则
- 处置建议
- 最终人工结论

## 4. 审核体系设计

## 4.1 前置审核：在生成前拦截风险

前置审核覆盖以下内容：

- 用户原始输入指令
- 补充说明文本
- 标签注入后的最终生成指令
- 学校上传的图片素材

前置审核的目标，是尽量把风险拦截在生成之前，减少无效算力消耗和成片后的返工成本。

执行策略如下：

- 文本输入使用 `腾讯云 TMS` 做同步审核
- 学校上传素材图片使用 `腾讯云 CI 图片审核`
- 平台本地再叠加“教育场景规则”

本地规则可以重点覆盖：

- 未成年人保护相关表述
- 真实学生/教师照片替换禁令
- 历史人物、政治表达、校园场景的特殊约束
- 学校自定义敏感词或禁止主题

## 4.2 后置审核：在成片后做最终把关

后置审核覆盖以下内容：

- 视频画面
- 字幕文本
- 配音音频
- 合成后的完整 MP4

后置审核采用“分层审核”：

1. 成片前先对字幕文本做同步审核
2. 对关键帧/代表帧做快速审核
3. 成片后再提交完整视频异步审核

这样做的价值在于：一方面可以降低整片被打回的概率，另一方面也能更早向教师暴露风险提示，提升流程效率。

## 4.3 同步审核与异步审核的拆分建议

平台从设计上明确区分同步审核与异步审核两条链路：

### 同步审核适合的场景

- 用户输入文本审核
- 脚本文本审核
- 单条字幕审核
- 图片上传审核
- 快速二次校验

同步审核适合用在“必须立刻给用户反馈”的节点，例如学生提交违规指令时，应立即提示“内容违规，请修改后重试”。

### 异步审核适合的场景

- 完整视频审核
- 批量素材审核
- 高峰期统一排队审核
- 需要回调和结果聚合的多模态审核

根据腾讯云官方资料，`数据万象 CI 的视频审核为异步任务模式`，更适合作为最终视频审核主通道；而 `TMS 文本审核` 则天然适合实时同步调用。因此本平台应设计成：

- 输入阶段：同步优先
- 输出阶段：异步优先
- 人工复审：事件驱动

## 4.4 腾讯云审核能力与成本考虑

结合腾讯云当前公开能力，平台可按以下方式接入审核服务：

- `TMS 文本审核`：按条计费，适合输入审核和字幕审核，适合实时拦截，成本相对稳定
- `CI 视频审核`：本质上由视频截帧、画面审核、音频审核等组成，适合成片审核，但单次成本显著高于文本审核
- `CI 视频审核` 为异步任务，不适合放在强实时链路里阻塞用户请求

从管理和成本角度，可以得出三个明确结论：

1. `越靠前的审核越便宜`，因为文本审核远低于视频审核成本。
2. `越完整的审核越耗时`，完整视频审核应进入队列和回调链路。
3. `MVP 阶段必须做“前置审核 + 后置审核”组合`，否则要么风险高，要么成本高。

从成本结构看，平台可以明确以下判断：

- 文本审核通常按“万条”计费
- 视频审核按截帧、图片审核、音频审核等组合消耗
- 资源包模式通常比纯按量更适合学校平台这种可预估业务

具体采购价格和资源包配置，可在实施阶段结合实际业务量和腾讯云正式报价进一步确认。

## 4.5 自定义审核规则设计

腾讯云审核策略支持通过 `BizType` 绑定不同审核策略，并支持自定义风险库、词库和阈值配置。这为平台构建分层规则提供了基础。平台侧同步建设自身规则中心，用于统一管理平台规则与学校规则。

规则分为三层：

1. 平台全局规则
   - 黄暴政恐等基础安全规则
   - 真人照片替换禁令
   - 未成年人场景强约束

2. 教育专项规则
   - 历史人物强制人工复审
   - 课堂适龄表达限制
   - 校园传播边界规则

3. 学校自定义规则
   - 学校敏感词
   - 校本资源白名单/黑名单
   - 本校默认审核阈值

平台侧统一整理审核结果格式，输出风险标签、命中片段、处置建议和人工复审结论，便于前端展示和后续审计。

## 5. 任务队列与高并发设计

### 5.1 任务队列的必要性

本项目的生成链路天然包含多个耗时环节：

- 脚本生成
- 多分镜素材生成
- 配音与字幕合成
- 视频拼接转码
- 视频审核

如果全部采用同步处理方式，系统在 50 到 200 并发区间内容易迅速失稳。因此，队列能力不是优化项，而是 MVP 阶段的基础能力。

### 5.2 任务队列拆分

任务队列至少拆成两类：

- `视频工作流队列`
  - 脚本生成
  - 分镜生成
  - 重绘画面
  - 重配音
  - 视频合成

- `审核任务队列`
  - 输入审核补偿任务
  - 图片/关键帧审核
  - 视频异步审核
  - 人工复审分发

这样拆分后可以实现：

- 生成高峰不会拖垮审核链路
- 审核高峰不会阻塞正常生成
- 可以分别扩容生成处理能力和审核处理能力

### 5.3 排队策略

- 按学校、角色、任务优先级做排队
- 学生任务默认优先级低于教师任务
- 重生成单镜头任务优先级高于新建整片任务
- 审核任务优先级高于低价值后台任务

同时补充以下控制能力：

- 任务超时控制
- 重复提交控制
- 失败重试机制
- 预计等待时间计算

这些能力将直接影响前端“排队位置”和“预计耗时”的展示准确性，也会影响用户对平台稳定性的感知。

## 6. 角色与权限实施方式

MVP 阶段采用统一后端，并通过角色、菜单和数据权限配置不同用户的可见范围与可操作范围。

后端侧按以下方式实施：

- 学生：开放任务发起、个人作品查看、退回原因查看
- 教师：开放脚本审核、镜头微调、学生作品审批、违规记录查看
- 学校管理员：开放学校素材管理、标签维护、学校范围内记录查看
- 平台运营：开放全局规则管理、异常任务处理、全局审计与运营配置

权限控制覆盖三层：

- 页面与菜单权限
- 接口操作权限
- 数据范围权限

输出效果上，保持同一套后端服务能力，但不同角色进入系统后看到的模块、可执行的动作和可查看的数据范围各不相同。

## 7. 部署与演进路径

## 7.1 MVP 阶段：Docker 部署

MVP 阶段采用：

- `Nginx`
- `FastAPI API`
- `异步任务处理服务`
- `PostgreSQL`
- `Redis`
- `MinIO/COS`
- `Prometheus/Grafana`

部署方式以 `Docker Compose` 或轻量容器编排为主，MVP 阶段重点是：

- 快速上线
- 容易排障
- 低运维门槛
- 支撑单校到多校早期试点

这一阶段的部署方式完全能够满足 MVP 的交付需求。

## 7.2 中期阶段：分布式服务与网关能力

当业务量上升到多学校、多批次高峰并发后，再逐步拆分为：

- 统一接入网关
- 用户与权限服务
- 工作流编排服务
- 生成服务
- 审核中台服务
- 资产中心服务
- 消息通知服务

这时的重点是：

- 服务独立扩容
- 不同模块独立发布
- 更精细的链路治理
- 统一日志与监控

## 7.3 后期阶段：K8s 化部署

当平台进入区域级或城市级部署时，再迁移到 `K8s` 更为合适。K8s 的价值主要在于：

- 弹性扩缩容
- 多环境标准化部署
- 灰度发布与回滚
- 更好的资源隔离

MVP 阶段暂不引入 K8s，优先保证第一阶段交付效率和部署稳定性。

## 8. 阶段任务交付列表

MVP 阶段按“每周或每个阶段都有可见成果”的方式推进。交付上不以一次性集中验收为主，而是逐步形成可查看、可联调、可验证的版本。

### 启动期交付（约 1 周）

这一阶段主要完成项目启动和基础准备，交付内容包括：

- MVP 范围确认结果
- 角色权限清单
- 主流程梳理结果
- 核心接口草案
- 核心数据模型
- 基础开发与测试环境
- 第三方服务接入验证结果

### 第一阶段交付（约 1-2 周）

这一阶段主要交付基础能力和主流程骨架，交付内容包括：

- 账号、组织、角色权限基础能力
- 任务中心基础能力
- 输入审核能力
- 脚本生成与脚本确认能力
- 基础任务状态流转
- 可供前端开始联调的核心接口

### 第二阶段交付（约 1-2 周）

这一阶段主要交付视频生成主链路，交付内容包括：

- 分镜素材生成能力
- 配音与字幕合成能力
- 视频合成能力
- 单镜头重生成能力
- 学校素材上传与调用能力
- 教师主流程可演示版本

### 第三阶段交付（约 1 周）

这一阶段主要交付审核与角色化能力，交付内容包括：

- 输出审核能力
- 人工复审流转能力
- 审核明细结果展示数据
- 违规日志与审计日志
- 学生、教师、学校管理员、平台运营的差异化权限能力
- 主要业务页面联调版本

### 第四阶段交付（约 1 周）

这一阶段主要交付稳定性和上线准备能力，交付内容包括：

- 功能测试结果
- 接口测试结果
- 关键链路回归结果
- 并发与排队校验结果
- 日志、监控与告警配置
- 预发环境部署版本

### 收尾阶段交付（约 1 周）

这一阶段主要完成问题修正和正式验收准备，交付内容包括：

- 联调问题修正版本
- 权限与规则调整版本
- 试运行结果
- MVP 验收版本
- 已知问题清单
- 后续阶段优化建议

## 8.1 MVP 阶段建议控制的范围

为了保证 MVP 阶段聚焦核心闭环，第一阶段聚焦以下必须项：

- 教师/学生/学校管理员/平台运营四类角色的最小权限闭环
- 输入审核、输出审核、人工复审闭环
- 教师脚本确认、单镜头重生成、成片入库闭环
- 学校资产上传与固定调用闭环
- 基础排队、状态展示、日志审计闭环

放到下一阶段的内容：

- 多人协作编辑
- 多模型智能路由
- 小程序端
- 复杂报表中心
- 跨区域多租户深度运营能力

这样更有利于保证第一版既能按期上线，也能真正被教师和学校使用，避免出现功能很多但主链路不稳定的问题。

## 8.2 后续阶段延展方向

在 MVP 稳定运行后，后续阶段可逐步扩展以下能力：

- 学校素材中心与标签体系增强
- 审核规则中心与运营配置增强
- 多学校并发调度和资源扩容
- 网关化与分布式服务拆分
- K8s 化部署与区域级运营支持
